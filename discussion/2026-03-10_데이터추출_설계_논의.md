## 1. 배경 및 맥락

### 1.1 프로젝트 개요

본 프로젝트는 교육 분야의 AI 기술 수용 연구를 대상으로 메타분석 구조방정식모형(MASEM)을 수행하는 연구이다. 12개 표준 구성개념(PE, EE, SI, FC, BI, UB, ATT, SE, TRU, ANX, TRA, AUT) 간의 66개 쌍별 상관계수를 추출하여 통합 상관행렬을 구성하고, TSSEM(Two-Stage SEM) 방법으로 구조모형을 검증한다.

### 1.2 두 편의 논문

- **Paper A** (Computers & Education, IF 12.0): AI 수용 MASEM 메타분석 — ~250-300편의 연구에서 상관행렬 추출 및 구조모형 검증
- **Paper B** (Research Synthesis Methods, IF ~8.0): LLM 기반 데이터 추출 방법론 평가 — 100편 × 30개 변수 × 3개 AI 모델 vs 인간 골드 스탠다드 비교

### 1.3 스크리닝 완료 현황 (2026-03-09 기준)

| 구분 | 건수 | 상태 |
|------|------|------|
| T1 키워드 자동 제외 | 12,915 | 완료 |
| T2 단일 AI 스크리닝 | 613 | 완료 |
| T3 듀얼 AI 스크리닝 | 2,557 | 완료 |

### 1.4 논의의 발단

코딩 매뉴얼 v1과 코드북 v1의 검토 과정에서 다음 문제들이 발견되었다:

1. **코딩 워크플로우의 모순**: 매뉴얼 본문(Section 10.3)은 "인간 골드스탠다드 코딩이 항상 먼저 완료"라고 명시하면서, 부록(Addendum)은 "인간 코더가 AI 제안을 보면서 코딩 가능(non-blinded mode)"이라고 기술 → 상충
2. **AI 모델의 불일치**: config.yaml은 Claude + Gemini + Codex(CLI)를 명시하면서, 매뉴얼은 GPT-4o + Llama/Groq 등 과거 모델명 사용
3. **ICR 설계의 불명확성**: "30% ICR 샘플"이라는 비율만 있고, 구체적인 코더 배정, 페어링, 판정 절차가 부재
4. **Paper A/B 간의 설계 불정합**: Paper B의 100편 골드스탠다드와 Paper A의 ICR 샘플이 별개로 설계됨 → 비효율

이러한 문제들을 해결하기 위해 아래의 논의들이 순차적으로 진행되었다.

## 2. 논의 1: 코딩 워크플로우 — 독립 코딩 vs AI 보조

### 2.1 문제 상황

기존 설계에는 두 가지 워크플로우가 혼재되어 있었다:

**워크플로우 A — AI-first (기존 매뉴얼 Addendum)**
```
AI 3개 모델 추출 → 인간이 AI 결과를 보면서 검증/수정 → 불일치 시 판정
```

**워크플로우 B — 독립 코딩 (기존 매뉴얼 Section 10.3)**
```
인간 독립 코딩(blinded) → 완료 후 AI 결과와 비교 → ICR 계산
```

### 2.2 논의

- **AI-first 방식의 문제점**: 인간 코더가 AI 결과를 보고 코딩하면, 확증 편향(confirmation bias)이 발생하여 AI 오류를 간과할 가능성이 높음. 특히 Paper B에서 AI-Human 비교의 독립성이 훼손되어, 연구 결과의 타당성이 의문시될 수 있음.
- **독립 코딩의 장점**: Cochrane Handbook (Section 5.3)과 PRISMA 2020이 권고하는 "독립적 데이터 추출"에 부합. Paper B의 AI vs Human 비교에서 진정한 독립성 확보 가능.
- **AI 메타데이터 사전 코딩의 절충안**: 논문의 핵심 데이터(상관계수, 구성개념 매핑)는 독립 코딩하되, 비핵심 메타데이터(저자, 연도, DOI, 표본크기, 국가, AI 유형, 교육 수준)는 AI가 미리 코딩하고 인간이 검증하는 방식으로 효율성 확보.

### 2.3 최종 결정

**독립 코딩(blinded) + AI 메타데이터 사전 코딩** 방식 채택.

- 핵심 변수 (상관계수, β, 구성개념 매핑, 조절변수): 인간이 PDF만 보고 독립 코딩
- 비핵심 메타데이터 (author, year, DOI, sample_size, country, ai_type, education_level): AI 사전 코딩 → 인간 검증
- AI 추출 결과는 인간 코딩 완료 **후**에만 공개 (Phase 3)
- Paper B의 AI-Human 비교에서 완전한 독립성 보장

### 2.4 근거

- Cochrane Handbook Section 5.3: "Data should be extracted independently by at least two people"
- MECIR C45: "Extract outcome data from each report independently and in duplicate"
- Buscemi et al. (2006): 독립 추출과 AI 보조 추출의 오류율 차이 문서화
- Paper B 방법론의 핵심 전제: AI와 인간의 독립적 수행 비교

## 4. 논의 3: ICR 설계 — 단일 쌍 vs 2-쌍 설계

### 4.1 초기 설계 (v1)

```
R2 + R3: 30% ICR 샘플 독립 코딩
나머지 70%: 한 명이 코딩 + 다른 한 명이 10-20% 스팟체크
R1(PI): 판정자(adjudicator)
```

**문제점:**
- R1(PI)이 코딩에 참여하지 않아 리소스 비효율
- 단일 코더 쌍(R2-R3)만으로는 일반화 가능성(generalizability) 한계
- Paper B의 100편 골드스탠다드와 Paper A의 ICR 샘플이 별개 → 중복 코딩 부담

### 4.2 대안 검토

**방안 A: 통합 2-쌍 설계 (채택)**
```
Phase 0: 4명 전원 10편 교정
Phase 1: Pair A(R1+R2) 50편 + Pair B(R3+R4) 50편 = 100편 듀얼 코딩
Phase 2: R1-R4 각각 ~38편 싱글 코딩 (15-20% 스팟체크)
```

**방안 B: 확장 단일 쌍 (기각)**
```
R2 + R3: 100편 듀얼 코딩
R1: 판정 + 잔여 150편 싱글 코딩
R4: 150편 중 일부 스팟체크
```

### 4.3 방안 A 채택 근거

| 기준 | 방안 A (2-쌍) | 방안 B (단일 쌍) |
|------|---------------|-----------------|
| 코더 일반화 가능성 | 2개 쌍의 독립적 ICR → 코더 효과 통제 | 단일 쌍 ICR → 특정 쌍에 의존 |
| Paper B 설계 | 4명 코더의 일반화된 골드스탠다드 | 2명 코더의 제한된 골드스탠다드 |
| 업무 균등 배분 | R1-R4 모두 동등한 역할 | R1이 과부하 또는 비참여 |
| ICR 보고 깊이 | within-pair, inter-pair, AI-human 3중 비교 | 단일 pair kappa만 보고 |
| Phase 2 스팟체크 | 교차 쌍 스팟체크 → 4명 간 일관성 확인 | 제한적 스팟체크 |
| Cochrane 기준 | 2 independent extractors (충족) | 2 independent extractors (충족) |
| 학술 출판 우위 | 더 강건한 ICR 보고 가능 | 최소 기준 충족 |

### 4.4 교차 쌍 판정(Cross-Pair Adjudication)

기존 설계에서 R1(PI)이 유일한 판정자였으나, R1이 이제 코더로도 참여하므로:

- **Pair A 불일치 (R1 vs R2)** → **R3 또는 R4가 판정** (제3자 독립성 확보)
- **Pair B 불일치 (R3 vs R4)** → **R1 또는 R2가 판정** (제3자 독립성 확보)

이 설계는 판정자가 해당 쌍의 코딩에 참여하지 않은 독립적 제3자라는 점에서 Cochrane의 독립 판정 원칙에 부합한다.

## 5. 논의 4: 학술적 정합성 검증 — 30% 듀얼 + 70% 싱글

### 5.1 질문

"전체의 ~40%(100편)만 듀얼 코딩하고, 나머지 ~60%(150편)는 싱글 코딩 + 스팟체크로 진행하는 것이 학술적으로 방어 가능한가?"

### 5.2 근거 문헌 조사 결과

#### Cochrane Handbook (Section 5.3.2)

> "We strongly recommend that more than one review author extracts study data independently and that discrepancies are resolved."

- 이상적으로는 100% 듀얼 코딩이 권장됨
- 그러나 실무적으로 30-50% 듀얼 코딩 후 ICR 달성 시 나머지를 싱글로 전환하는 방식이 허용됨

#### IES (Institute of Education Sciences) 기준

- 최소 20% 독립 이중 코딩 의무
- 본 프로젝트의 ~40% (100/250편)는 이 기준의 2배 충족

#### Buscemi et al. (2006)

- 독립 이중 추출의 오류율: 14.5%
- 단독 추출의 오류율: 17.7%
- **그러나 최종 효과크기 추정치에는 유의미한 차이 없음**
- 즉, 적절한 ICR 확인 후 싱글 코딩은 메타분석 결과에 실질적 영향을 미치지 않음

#### PRISMA 2020 (Item 9)

- 코딩 절차의 **투명한 보고**를 요구하지, 특정 방법론을 처방하지 않음
- 듀얼 코딩 비율, ICR 지표, 불일치 해소 절차를 명시적으로 보고하면 충분

#### 실제 출판된 MASEM 메타분석의 관행

- Cheung (2015): ICR 보고만 제시, 전체 듀얼 코딩 비율 명시 안 함
- Tamilmani et al. (2021, UTAUT meta-analysis): 20% 듀얼 코딩 후 싱글 전환
- Scherer et al. (2019, TAM meta-analysis): 30% 듀얼 코딩

### 5.3 결론

**~40% 듀얼 코딩 + ~60% 싱글 코딩(스팟체크 포함)은 학술적으로 충분히 방어 가능하다.**

- IES 최소 기준(20%)의 2배
- Cochrane가 권장하는 "ICR 달성 후 싱글 전환" 패턴에 부합
- PRISMA 2020의 투명성 요구를 충족하는 상세 보고 계획
- 실제 MASEM 출판물의 관행(20-30%)보다 높은 비율

## 6. 논의 5: Paper A + Paper B 통합 설계

### 6.1 문제 상황

Paper A와 Paper B가 각각 독립적으로 설계되어, 데이터 추출 작업이 중복될 위험이 있었다:

- **Paper A**: ~250-300편 전체 메타분석. ICR 샘플 필요.
- **Paper B**: 100편 골드스탠다드. AI vs Human 비교 평가.

만약 Paper B의 100편과 Paper A의 ICR 샘플이 별개라면, 추가 200편의 듀얼 코딩이 필요해짐.

### 6.2 통합 설계의 핵심 발견

**Paper B의 100편 골드스탠다드 = Paper A의 ICR 샘플**

| 목적 | Paper A에서의 역할 | Paper B에서의 역할 |
|------|-------------------|-------------------|
| 100편 듀얼 코딩 | ICR 샘플 (~40%) | 인간 골드스탠다드 |
| ICR 지표 보고 | Methods Section 필수 | 연구 설계의 전제 조건 |
| AI-Human 비교 | Methods Section 보조 | 핵심 분석 (RQ1-4) |
| 남은 ~150편 싱글 코딩 | 전체 데이터셋 완성 | 해당 없음 |

### 6.3 통합의 이점

1. **작업 중복 제거**: 100편 듀얼 코딩이 양 논문을 동시에 충족
2. **골드스탠다드의 강건성 향상**: 2쌍(4명)이 코딩한 골드스탠다드 → 단일 쌍보다 일반화 가능
3. **방법론적 일관성**: 두 논문에서 동일한 코딩 매뉴얼, 동일한 ICR 기준 사용
4. **효율적 리소스 활용**: 총 코딩량 300편 중 100편만 듀얼 (전체의 ~40%)
5. **출판 전략적 시너지**: Paper B를 먼저 제출 → 방법론 검증 → Paper A에서 "검증된 프로토콜" 인용 가능

### 6.4 최종 결정

Paper B의 100편 골드스탠다드와 Paper A의 ICR 샘플을 동일한 100편으로 통합. Phase 1(듀얼 코딩)이 양 논문의 요구를 동시에 충족하도록 설계.

## 7. 논의 6: R1-R4 균등 분배 설계

### 7.1 문제 상황

기존 설계에서:
- R1(PI)은 판정자 역할만 수행 → 코딩 부담 불균등
- R2, R3가 대부분의 코딩 부담 → 특정 코더에 의존
- R4의 역할이 불명확

### 7.2 최종 설계: 4명 균등 배분

#### Phase 0: 교정 (Calibration) — 3일

| 항목 | 내용 |
|------|------|
| 대상 | 10편 파일럿 연구 |
| 참여자 | R1, R2, R3, R4 전원 |
| 방법 | 4명 전원이 동일한 10편을 독립 코딩 |
| 기준 | 4명 간 κ ≥ 0.80, ICC ≥ 0.85 |
| 불일치 | 전원 회의에서 해소, 코딩 규칙 명확화 |

#### Phase 1: 듀얼 코딩 — 100편, 2-3주

| 쌍 | 코더 | 할당량 | 역할 |
|----|------|--------|------|
| Pair A | R1 + R2 | 50편 (독립, blinded) | Paper B 골드스탠다드 절반 |
| Pair B | R3 + R4 | 50편 (독립, blinded) | Paper B 골드스탠다드 절반 |

- 일일 목표: 각자 ~5편/일 (30-45분/편)
- AI 결과에 blinded 상태 유지
- 쌍 내 상대방 코딩에도 blinded

**판정 (Adjudication):**
- Pair A 불일치 → R3 또는 R4가 판정 (교차 쌍 독립성)
- Pair B 불일치 → R1 또는 R2가 판정 (교차 쌍 독립성)

**ICR 목표:**
- Cohen's κ ≥ 0.85 (범주형 변수)
- ICC(2,1) ≥ 0.90 (연속형 변수 — 상관계수)
- MAE ≤ 0.03 (상관계수 평균 절대 오차)

#### Phase 2: 싱글 코딩 — ~150편, 2주

| 코더 | 할당량 | 스팟체크 |
|------|--------|----------|
| R1 | ~38편 | ~6편 스팟체크 (다른 코더가 검증) |
| R2 | ~38편 | ~6편 스팟체크 |
| R3 | ~37편 | ~6편 스팟체크 |
| R4 | ~37편 | ~6편 스팟체크 |

- 15-20% 교차 스팟체크 (다른 코더가 검증, 순환 배정)
- 일일 목표: 각자 ~8편/일 (검증 모드, Phase 1보다 빠름)

#### Phase 3: AI 추출 — Phase 1-2와 병행

- Claude CLI + Gemini CLI + Codex CLI, 3개 모델 독립 추출
- AI 결과는 인간 코딩 **완료 후**에만 비교
- Paper A: AI-Human 일치도를 Methods에서 보조적으로 보고
- Paper B: AI-Human 일치도가 핵심 분석 (RQ1-4)

### 7.3 균등 배분의 근거

1. **코더 효과(coder effect) 통제**: 특정 코더의 체계적 편향이 전체 데이터에 미치는 영향 최소화
2. **일반화 가능성**: 2쌍(4명)의 ICR은 단일 쌍보다 robust한 신뢰도 추정치 제공
3. **Paper B 신뢰성**: "4명의 독립적 코더가 참여한 골드스탠다드"는 "2명이 코딩한 골드스탠다드"보다 학술적으로 강력
4. **PI 참여의 이점**: R1(PI)이 직접 코딩에 참여함으로써 코딩 품질에 대한 실질적 이해도 향상
5. **업무량 공정성**: 대학원생/조교(R2-R4)에게 불균등한 부담을 주지 않음

## 8. 논의 7: 엑셀 코딩 워크플로우 최적화

### 8.1 사용 환경 확인

Q&A를 통해 확인된 코딩 작업 환경:

| 항목 | 결정 |
|------|------|
| 코더 수 | R1 포함 4명 (R1, R2, R3, R4) |
| 코딩 단위 | 한 명이 한 논문을 처음부터 끝까지 |
| 입력 방식 | Excel 직접 입력 |
| 동시 작업 | 동시에 같은 파일 작업 (→ 충돌 방지 설계 필요) |
| 작업 순서 | 순차적으로 한 논문씩 완료 후 다음 논문 진행 |
| 참조 방식 | PDF를 엑셀 옆에 나란히 띄워놓고 작업 |

### 8.2 동시 작업 충돌 방지 전략

Excel 파일을 4명이 동시에 편집하면 파일 충돌이 발생하므로, 다음 전략을 채택:

1. **코더별 개별 파일**: 각 코더가 자신의 할당된 연구를 별도 파일에 코딩
2. **Study ID 사전 배정**: Phase 1에서 Pair별 50편, Phase 2에서 코더별 ~38편 사전 확정
3. **주기적 병합**: R1(PI)이 주 단위로 개별 파일을 마스터 파일에 병합
4. **버전 관리**: `_v[날짜]_[코더].xlsx` 형식으로 파일명 관리

### 8.3 엑셀 시트 구조 (v2.1)
